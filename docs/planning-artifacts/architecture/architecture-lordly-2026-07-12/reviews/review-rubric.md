# Rubric Review — Architecture Spine, Lord Battle Tactics

**Verdict: PASS WITH FINDINGS.** The spine is fit for purpose as a fence for AI codegen: the paradigm is crisp, the ADs target real divergence points, most Rules are mechanically enforceable, and terseness is used well. One Major finding (seed-stream architecture) sits directly on the project's core invariant (determinism, FR20) and should be fixed before epic breakdown. The remaining findings are Moderate/Minor tightening.

Context honored: hobby project, solo creator, AI-generated code via BMAD, purpose=build-substrate. Enforceability weighted above completeness of prose.

---

## Checklist Item 1 — Fixes the real divergence points, misses none

**Mostly yes.** The eight ADs hit the genuine cross-unit fault lines for AI-built stories: core/shell boundary (AD-1), log-replay contract (AD-2), dependency direction (AD-3), single vocabulary owner (AD-4), single match-state owner (AD-5), AI purity (AD-6), one platform (AD-7), one persistence scheme (AD-8). The Consistency Conventions table catches the second tier well (event shape, unit identity, error posture, mutation discipline).

### Finding R1 (MAJOR) — Seed-stream architecture is gestured at, not fixed

The PRD splits randomness across at least four consumers with different lifetimes and visibility rules:

- **Player element rolls** (FR3) — happen at *draft time*, in the shell's timeline, visible to the owner immediately, before `resolveBattle` is ever called.
- **AI element rolls** (FR3 + FR24) — must be hidden until reveal.
- **AI strategy selection** (FR25) — "seeded variation", AD-6 takes "(strategy pool, seed)".
- **Battle-resolution randomness** (FR13 coin flips, FR16 confusion) — inside `resolveBattle(setup, seed)`.

The spine hints at the answer in three places — `engine/rng.ts` "stream derivation" in the Structural Seed, the Capability Map row "element roll uses the match seed (AD-1)", and AD-6's separate seed parameter — but **no AD fixes who derives which stream from what, or whether the draft-time element roll and the battle seed are the same seed, derived streams, or independent values.** This is exactly the kind of question two independently-built stories (Draft story vs. Battle story vs. AI story) will answer differently, and every wrong answer silently breaks FR20 determinism or FR24 hiddenness — the two invariants the whole architecture exists to protect. It also touches AD-8: history "stores the full MatchSetup, seed" — if elements are re-derivable from the seed, storing them in MatchSetup is redundant-but-safe; if they are not, an agent that stores only the seed produces unreplayable history.

**Recommendation:** promote to an AD (or a Consistency Convention row): one match seed generated by the shell (already stated in conventions); the engine exports named derived streams (e.g., `elements:player`, `elements:ai`, `ai:strategy`, `battle`) via `rng.ts`; the shell never constructs randomness for domain purposes; `MatchSetup` carries the rolled elements explicitly so replay never re-rolls. Four sentences close the hole.

### Finding R2 (MODERATE) — AD-5's scene enumeration omits two PRD-required screens

AD-5 rules "each screen (Home, Draft, Placement, Reveal, Battle, Result, History) is a Phaser Scene" — presented as the exhaustive set, and repeated verbatim in the Structural Seed. But the PRD requires two more surfaces: the **rules/help screen** reachable from Home and Draft (FR27) and the **credits/attribution screen** (FR31). An agent building those has no answer for whether they are Scenes in the FSM, DOM overlays, or modals — and whether they receive `MatchState`. Given AD-5's rule is otherwise strict ("passed explicitly on every scene transition"), the omission invites ad-hoc divergence. Add them to the list (or explicitly rule them out-of-FSM overlays).

### Finding R6 (MINOR) — Asset/animation conventions unaddressed

FR31 requires idle/attack-cast/hurt/death representations per class, consumed by at least Draft (sprites on cards) and Battle (animations). The Deferred entry correctly defers *which pack*, but the naming/lookup convention (class → animation-key mapping, where the presentation lookup table lives) is a cross-story concern with no owner. Low blast radius (unit-identity convention already pushes sprites to presentation-layer lookups), but one convention row would prevent two scenes inventing two lookup schemes.

## Checklist Item 2 — Every Rule enforceable and divergence-preventing

**Strong overall.** Best-in-class here: AD-1 (purity + single-dependency whitelist — mechanically checkable), AD-3 (workspace dependency direction — lintable), AD-4 (single type owner), AD-6 (hiddenness "enforced by construction, not discipline" via function signature — exactly right for AI codegen). AD-2's "anything the UI needs must be an event in the log" is a clean litmus test.

### Finding R3 (MODERATE) — AD-8 / balanceVersion bump has no enforcement mechanism

The whole replayability story (AD-2 + AD-8) hinges on `balanceVersion` being bumped whenever `balance.ts` data changes. The convention declares it "monotonic integer" but nothing enforces the bump — no test, no CI check, no rule. An AI agent tuning stats (which the PRD says will happen constantly, FR15 note, NFR4) that forgets the bump makes old history replay *wrong* battles silently — worse than the non-replayable case AD-8 guards against. Cheap fix: a golden test that snapshots balance data keyed by version, failing when data changes under an unchanged version. One sentence in the test conventions row makes this enforceable.

Minor enforceability notes:
- AD-7 "red blocks merge" — for a solo dev pushing to `main`, this needs branch protection or a deploy-gated-on-tests workflow ordering to be real; the rule as written is enforceable, just worth wiring correctly in the epic.
- NFR2's assumed ≥90% line-coverage gate on the engine appears nowhere in AD-7 or the test conventions — either adopt it into the CI rule or explicitly reject the PRD assumption. Silent drop of a bound NFR clause. (Folded into R5.)

### Finding R4 (MODERATE) — NFR3 unmapped and mostly ungoverned; NFR6's army-size parameter never stated

- **NFR3 (documentation as first-class)** requires a README, a generated/aligned rules document, ADRs, and doc comments on the engine's public API. The map has no row for it; the conventions table has no docs row; only the rules-doc *generation mechanism* appears (in Deferred). For a project whose PRD calls documentation "survival," the spine is silent on where docs live and what an agent must produce per story. One map row + one convention line fixes it.
- **NFR6** explicitly requires "army size is a parameter (3 now, 5-slot era later)." The spine binds NFR6 (frontmatter, link-play map row) but never states the parameterization — nothing stops an AI agent baking `3` into `MatchSetup` types, tuple-typed armies, or a 3-hardcoded targeting loop. This is precisely a by-construction rule the spine should carry (one clause in AD-1 or AD-4).
